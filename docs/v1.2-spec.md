# Localzada v1.2 — Technical Specification

**Status:** Draft for review
**Author:** Mohammed (Ai RawayieAldhaka)
**Last updated:** 2026-04-27
**Scope:** Personal Pro tier (single user, single CF account)

---

## 1. Goals & non-goals

### Goals
- A `localzada dev <profile>` command that spins up a multi-process dev environment defined declaratively.
- Per-project config via `localzada.yaml` with sensible global defaults from `~/.config/localzada/config.json`.
- Named Cloudflare Tunnels with stable subdomains for **webhook testing** (Stripe, Telegram bots like QanunAI).
- Quick Tunnels remain the default for ad-hoc shares.

### Non-goals (deferred to v1.3+)
- Multi-user / team profiles.
- License server, Stripe billing, JWT validation.
- Hot-reload of `localzada.yaml` while `dev` is running.
- Auto-download of cloudflared binary.

### Explicit anti-goals
- No process supervision/restart-on-crash beyond surfacing the failure. Localzada is not a process manager replacement; it's a dev orchestrator. If a child crashes, we report it — we don't loop-restart. *(Open question — see §11.)*

## 2. The two real use cases (driving the design)

Every decision in this spec must serve at least one:

### UC-1: Multi-project dev environment
Mohammed has 4 active projects (QanunAI, WhatsApp suite, AI tools ebook backend, Multi-AI Hub). Each has its own ports, env vars, and run command. Today he opens 4 terminals and copies commands from notes.

**With v1.2:**
```
$ cd ~/projects/qanunai
$ localzada dev
✓ api          → http://localhost:8000  (uvicorn)
✓ telegram-bot → http://localhost:8001  (python bot.py)
✓ tunnel       → https://qanunai.example.dev  (named)
```

### UC-2: Webhook testing
Telegram requires a public HTTPS URL to deliver updates to QanunAI. Stripe requires the same for payment events. The URL must be **stable** — Telegram caches webhook URLs, and updating it on every dev restart is friction.

**With v1.2:**
```yaml
# localzada.yaml
profile: qanunai
tunnel:
  type: named
  hostname: qanunai-dev.example.dev   # stable, registered on CF once
  service: http://localhost:8000
```

`qanunai-dev.example.dev` resolves to your laptop today, your other laptop tomorrow, your colleague's machine on demo day. Telegram never knows.

## 3. Configuration model — the layered approach

Two files. **Project wins where they overlap.**

### 3.1 Global: `~/.config/localzada/config.json`
Already exists in v1.1 (managed by `conf` package). v1.2 extends it.

```json
{
  "plan": "free",
  "lang": "ar",
  "telemetry": false,
  "tunnel": {
    "provider": "cloudflare",
    "defaultType": "quick",
    "cloudflare": {
      "accountId": "<from cloudflared login>",
      "defaultZone": "example.dev"
    }
  },
  "defaults": {
    "logRetentionDays": 7,
    "killGracePeriodMs": 5000
  }
}
```

**Purpose:** machine-wide defaults, CF account binding, things you don't want to repeat per project.

### 3.2 Project: `./localzada.yaml`
New in v1.2. Lives at the repo root. Discovered by walking up from `cwd` (like git does for `.git`).

```yaml
# localzada.yaml — QanunAI example
version: 1
profile: qanunai

# Multiple processes, run in parallel
services:
  api:
    cmd: uvicorn main:app --reload --port 8000
    port: 8000
    env:
      DATABASE_URL: postgresql://localhost/qanunai_dev
      ANTHROPIC_API_KEY: ${ANTHROPIC_API_KEY}   # from shell env
    waitFor: tcp                                # health check
    healthTimeout: 30s

  telegram-bot:
    cmd: python bot.py
    dependsOn: [api]                            # api must be healthy first
    env:
      TELEGRAM_TOKEN: ${TELEGRAM_TOKEN}
      WEBHOOK_URL: https://qanunai-dev.example.dev/webhook

# Tunnel config (optional)
tunnel:
  type: named                                   # or "quick"
  hostname: qanunai-dev.example.dev
  service: http://localhost:8000                # which local port to expose
  startWith: api                                # tunnel opens after `api` is healthy
```

### 3.3 Merge precedence (decided)

```
shell env vars  >  project localzada.yaml  >  global config.json  >  built-in defaults
```

For nested objects we do **deep merge** (project overrides individual keys, doesn't replace whole sections). Arrays are **replaced wholesale** — no item-level merging, because YAML arrays don't have stable identity.

### 3.4 Discovery algorithm

```
1. From process.cwd(), walk upward looking for localzada.yaml
2. Stop at: filesystem root, OR a directory containing .git, OR a directory
   containing package.json — whichever comes first (these are project-root signals)
3. If found: load + merge with global config
4. If not found: use global config alone
5. `localzada dev <profile>` without a yaml → error: "no localzada.yaml found"
```

The `.git`/`package.json` boundary prevents accidentally picking up a `localzada.yaml` from a parent directory that isn't actually the project root.

## 4. New commands in v1.2

### 4.1 `localzada dev [profile]`

The flagship command. Reads `localzada.yaml`, spawns all `services` in dependency order, optionally opens the tunnel, streams unified logs to stdout.

```bash
localzada dev                  # uses profile from yaml
localzada dev --detach         # background mode (returns immediately)
localzada dev --no-tunnel      # skip tunnel even if defined
localzada dev --only api       # run a subset
```

**Behavior:**
- **Foreground mode (default):** unified colored log stream, prefixed by service name. `Ctrl+C` stops everything in reverse dependency order.
- **Detach mode:** spawns everything as in v1.1's `start`, returns immediately. `localzada status` shows the group.
- All services and the tunnel are tagged with the same `groupId` so `localzada stop --group qanunai` stops them together.

### 4.2 `localzada tunnel <subcommand>`

Replaces / extends `share` with named-tunnel awareness.

```bash
localzada tunnel quick --port 3000               # = old `share`
localzada tunnel up qanunai-dev.example.dev      # named tunnel from yaml
localzada tunnel down [hostname]                  # stop a tunnel
localzada tunnel list                             # show all named tunnels on this CF account
localzada tunnel route <hostname> <port>          # one-off DNS route
```

**Backward compat:** `localzada share --port X` still works, calls `tunnel quick` internally.

### 4.3 `localzada init`

Bootstrap a new `localzada.yaml` interactively or from a template.

```bash
localzada init                    # interactive prompts
localzada init --template fastapi # use a built-in template
localzada init --from package.json  # infer ports/scripts from npm scripts
```

### 4.4 `localzada login`

Wraps `cloudflared tunnel login`. Stores the path to the cert in global config so we don't re-prompt. Required once before any named tunnel ops.

## 6. The `services` orchestrator (the hard part)

This is the one piece that doesn't exist in v1.1 at all. Treat as the highest-risk component.

### 6.1 Dependency graph
- Each service may declare `dependsOn: [other-service]`.
- We compute topological order. Cycles → fail with a clear error before spawning anything.
- Services in the same level start in parallel.

### 6.2 Health checks (`waitFor`)
Three modes:
- `none` (default if omitted): start the next service immediately after spawn.
- `tcp`: wait for the declared `port` to accept connections (uses v1.1's `waitForPort`).
- `http`: GET the declared health URL until 2xx (v1.2 introduces this).

`healthTimeout` defaults to 30s. Exceeding it fails the whole `dev` and tears down everything started so far.

### 6.3 Failure model
- If service A fails to start: services that already started keep running (so you can debug), `dev` exits with non-zero.
- If service A crashes mid-run during foreground mode: print red banner, leave the rest running, do NOT auto-restart. User decides.
- If `Ctrl+C`: send SIGTERM to all in **reverse** dependency order, wait `killGracePeriodMs`, then SIGKILL stragglers.

### 6.4 Log multiplexing
Foreground mode prints lines like:
```
[api]          INFO: Application startup complete.
[telegram-bot] Polling started for @qanunai_bot
[tunnel]       https://qanunai-dev.example.dev → localhost:8000
```

Each service has a stable color (hashed from name). The prefix width is the longest service name + 1.

## 7. Named tunnels — implementation notes

### 7.1 Setup flow (one-time per machine)
```
$ localzada login
Opens browser for cloudflared tunnel login → produces ~/.cloudflared/cert.pem.
We store the cert path in global config.

$ localzada tunnel up qanunai-dev.example.dev --service http://localhost:8000
Behind the scenes:
  1. cloudflared tunnel create qanunai-dev          (if not exists)
  2. cloudflared tunnel route dns <id> qanunai-dev.example.dev
  3. Write ~/.cloudflared/<id>.json credentials (cloudflared does this)
  4. Spawn cloudflared tunnel run <id> with config in ~/.localzada/tunnels/<id>.yaml
```

We persist the tunnel ID + hostname in `~/.localzada/tunnels.json` so we can find it again without round-tripping CF API.

### 7.2 Why we don't use the CF API directly
`cloudflared` already speaks to CF and handles cert/credentials. Re-implementing in Node would mean:
- Storing CF API tokens locally (more attack surface).
- Re-doing DNS record management (which `cloudflared tunnel route dns` does atomically).

We delegate. v1.2 = thin wrapper around `cloudflared` subcommands.

### 7.3 Quick vs named — when each is used

| Need | Use | Example |
|---|---|---|
| Show a colleague my dev server right now | Quick | `localzada tunnel quick --port 3000` |
| Stripe webhook for development | Named | `localzada tunnel up stripe-dev.example.dev` |
| Telegram bot webhook | Named | `localzada tunnel up qanunai-dev.example.dev` |
| Demo to a client (URL in calendar invite) | Named | preferable — Quick URLs change every restart |
| Quick test in a Slack thread | Quick | URL only needs to work for an hour |

Default in `localzada.yaml` if `tunnel:` block exists but `type:` is missing → **named** (because that's the case where you bothered to write a config file). Default for bare `localzada share --port X` → **quick**.

## 8. Pro features (personal Pro, not yet enforced)

The plan field in global config can be `pro` for now. **No license validation in v1.2.** That's intentional — we're discovering shape. Enforcement comes in v1.3 when we add a license server.

### Free tier limits

- Up to **3 services** per `localzada.yaml` profile.
- Up to **2 service groups** running concurrently across the system.
- **Quick Tunnels only** (`tunnel.type: quick` or no `tunnel:` block).
- All v1.1 commands remain free: `start`, `stop`, `share`, `status`, `logs`, `doctor`, `config`.

### Pro tier limits

- Up to **5 services** per profile.
- Up to **5 service groups** running concurrently.
- **Named Tunnels** (`tunnel.type: named` with stable hostname).
- Multiple named tunnels per machine (CF account permitting).

### Rationale for the 3/5 split

The numbers are chosen against Mohammed's real workloads, not arbitrary marketing tiers:

- **QanunAI today** = api + worker + telegram-bot = 3 services. Stays in free.
- **QanunAI + named tunnel for Telegram webhooks** = 3 services + named tunnel → Pro is required for the tunnel, not for the service count. The tunnel is the meaningful gate.
- **Future QanunAI** (api + worker + bot + frontend + monitoring) = 5 services. Fits in Pro.
- **Beyond 5** = either you're running unrelated projects (use multiple groups) or you've outgrown personal-Pro shape entirely (different conversation).

This means: **the count limits almost never bite in practice; the named-tunnel gate is the real Pro signal.** Intentional. Counting services as a paywall is friction; gating a feature that requires a CF account anyway is honest.

### What this means operationally

A user upgrading from free to pro in v1.2 sees:
1. Their existing `localzada.yaml` keeps working (3 services or fewer).
2. They unlock named tunnels — the actual reason most people will care.
3. They can grow to 5 services without rewriting their config.

This split keeps the free experience identical to v1.1 — no regressions for existing free users.

## 9. File system layout (after v1.2)

```
~/.config/localzada/
  config.json                      # global config (managed by `conf`)

~/.localzada/
  sessions.json                    # v1.1 — running sessions registry
  tunnels.json                     # v1.2 — named tunnel registry (id → hostname mapping)
  groups/                          # v1.2 — group metadata
    qanunai.json
    whatsapp-suite.json
  logs/
    <session-id>.log               # v1.1
    <group-name>-<service>.log     # v1.2
  bin/                              # reserved for v1.3 auto-download
  *.lock                            # v1.1 lock files

~/.cloudflared/                     # managed by cloudflared, not us
  cert.pem
  <tunnel-id>.json
```

We do not move or symlink anything that `cloudflared` owns. We only read.

## 10. Backward compatibility contract

A user upgrading from v1.1 to v1.2 must observe:
1. All v1.1 commands work identically. `share`, `start`, `stop`, etc. unchanged in CLI surface.
2. v1.1 sessions.json is forward-compatible (no schema break — only additive fields).
3. No `localzada.yaml`? No new behavior. v1.2 reduces to v1.1 cleanly.

If we ever need to break sessions.json shape, v1.2 includes a migration path that reads the old shape and rewrites it. Not expected to be needed.

## 11. Open questions (must be resolved before implementation)

These are decisions Mohammed needs to make. Building any of these wrong wastes a week.

1. **Auto-restart on crash?** Spec currently says no. But for Telegram webhook testing, a crashed bot loses messages. Suggested: per-service `restart: on-failure` flag, default off. Risk: implements 30% of pm2 — feature creep.

2. **Tunnel persistence across reboots?** A named tunnel's `cloudflared tunnel run` process dies when laptop sleeps. Should `localzada dev` re-establish it on next run automatically (yes) or require explicit `tunnel up` (no)? Suggested: yes, automatic.

3. **YAML or TOML for `localzada.yaml`?** YAML is more common for dev tools but has whitespace footguns. TOML is what Cargo/poetry use. Suggested: YAML, because env var interpolation `${VAR}` reads more naturally and most JS devs know YAML from CI configs.

4. **`localzada init --from package.json`** — how aggressive is the inference? Reading `npm run dev` is easy. Detecting that this project has both a Next.js frontend and an Express backend on different ports is hard. Suggested: v1.2 only handles the trivial single-script case. Punt the smart inference to v1.3.

5. **Are env vars from `.env` files auto-loaded?** Spec currently doesn't say. The right answer is probably "yes, look for `.env` in the project root and `.env.local` if present, like Next.js does." Risk: collisions when a service has its own .env.

6. **What happens if `cloudflared` is logged in to a CF account that doesn't own the zone in `localzada.yaml`?** We must detect this before spawning and give a clear error: "Your cloudflared session is bound to account X, but `qanunai-dev.example.dev` belongs to zone Y in account Z."

## 13. Risk register

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| `cloudflared` API changes between minor versions | Medium | High | Pin minimum version in doctor; smoke test on every cloudflared update |
| YAML schema drifts as we use it | High | Medium | Version field + clear migration policy from day 1 (`version: 1`) |
| Service orchestrator becomes pm2 in disguise | Medium | High | Hard rule: no auto-restart in v1.2. Re-evaluate in v1.3. |
| Named tunnel creation fails halfway, leaving orphaned CF resources | Medium | Medium | Always clean up on failure; `localzada tunnel list` shows orphans |
| Users assume Pro features are licensed when they aren't | Low | Low | Doctor output always shows current plan + how it was set |
